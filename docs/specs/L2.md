# L2 - Detailed Requirements

## L2-001: Section Table Schema
**Traces to:** L1-001

The database must contain a `section` table with the following columns: `id` (INT IDENTITY, primary key), `parent_id` (INT, nullable, foreign key to `section.id`), `title` (NVARCHAR(200), NOT NULL), and `position` (INT, NOT NULL). A NULL `parent_id` denotes a root-level section.

**Acceptance Criteria:**
1. Given a fresh database, when migrations are applied, then the `section` table exists with exactly the columns `id`, `parent_id`, `title`, `position` and the specified types and nullability.
2. Given the `section` table, when a row is inserted with `parent_id` referencing a non-existent section, then the insert must fail with a foreign key violation.
3. Given the `section` table, when a row is inserted with `title` longer than 200 characters, then the insert must fail.

---

## L2-002: Fact Table Schema
**Traces to:** L1-001

The database must contain a `fact` table with the following columns: `id` (INT IDENTITY, primary key), `section_id` (INT, NOT NULL, foreign key to `section.id`), `text` (NVARCHAR(MAX), NOT NULL), and `position` (INT, NOT NULL).

**Acceptance Criteria:**
1. Given a fresh database, when migrations are applied, then the `fact` table exists with exactly the columns `id`, `section_id`, `text`, `position` and the specified types and nullability.
2. Given the `fact` table, when a row is inserted with `section_id` referencing a non-existent section, then the insert must fail with a foreign key violation.
3. Given the `fact` table, when a row is inserted with an empty string for `text`, then the insert must fail (CHECK constraint `LEN(text) > 0`).

---

## L2-003: Cascading Delete
**Traces to:** L1-005

Deleting a section must remove all descendant sections and all facts under those sections in a single atomic operation, enforced by the database engine.

**Acceptance Criteria:**
1. Given a section with two child sections and five facts spread among them, when the parent section is deleted, then all child sections and all five facts no longer exist in the database after the operation commits.
2. Given a section delete that fails midway due to a transient error, when the transaction rolls back, then the section, its descendants, and its facts must remain intact and unchanged.

---

## L2-004: Sibling Ordering by Position
**Traces to:** L1-005

Sibling rows (sections sharing a `parent_id`, or facts sharing a `section_id`) must be ordered by `position` ascending on every read. The application layer is responsible for assigning `position` values; the database must not assume contiguity but must support deterministic sort.

**Acceptance Criteria:**
1. Given three sibling sections with `position` values 10, 20, 30, when the children of their parent are queried ordered by `position`, then they are returned in that order.
2. Given three sibling facts with `position` values 5, 1, 3, when they are queried ordered by `position`, then they are returned in the order 1, 3, 5.
3. Given two siblings with the same `parent_id` and the same `position` value, when queried, then the result order may be non-deterministic; the application is responsible for never assigning duplicate positions among siblings.

---

## L2-005: Indexes Supporting Tree Traversal
**Traces to:** L1-003

The database must include the following non-clustered indexes to support fast tree assembly: an index on `section(parent_id, position)` and an index on `fact(section_id, position)`.

**Acceptance Criteria:**
1. Given the database after migrations, when system catalog views (`sys.indexes`) are queried, then both indexes exist with the specified key columns in the specified order.
2. Given a tree of 1,000 sections and 10,000 facts, when the full tree is loaded by the application using these indexes, then the database execution plan must show index seeks (not table scans) for both reads.

---

## L2-006: Full-Tree Read Latency
**Traces to:** L1-003

The database must service a full-tree read (all sections and all facts in two queries) within a strict latency budget appropriate for interactive editing.

**Acceptance Criteria:**
1. Given a tree of 1,000 sections and 10,000 facts on the target Azure SQL tier, when the application issues `SELECT id, parent_id, title, position FROM section` and `SELECT id, section_id, text, position FROM fact`, then both queries combined must complete in under 100 ms at the p95.
2. Given the same dataset, when the same queries are issued from a client co-located in the same Azure region as the database, then the round-trip p99 must be under 250 ms.

---

## L2-007: Single-Row Write Latency
**Traces to:** L1-003

Any single insert, update, or delete against `section` or `fact` must complete within an interactive latency budget.

**Acceptance Criteria:**
1. Given an existing tree and a warmed connection pool, when the application inserts a single fact, then the operation must complete in under 50 ms at the p95.
2. Given the same conditions, when the application updates a single fact's `text`, then the operation must complete in under 50 ms at the p95.
3. Given the same conditions, when the application deletes a single fact, then the operation must complete in under 50 ms at the p95.

---

## L2-008: Azure SQL Database Tier and Always-On Compute
**Traces to:** L1-004

The database must be deployed as an Azure SQL Database (single database, not Managed Instance) on a tier that does not auto-pause and does not impose cold-start latency on first request.

**Acceptance Criteria:**
1. Given the deployed database, when its configuration is inspected, then the service objective must be a provisioned tier (e.g., General Purpose Gen5 with a small vCore count) OR a Serverless tier with auto-pause explicitly disabled.
2. Given the database has been idle for one hour, when a client issues its first query, then the response must arrive in under 500 ms (i.e., no resume-from-pause delay).

---

## L2-009: Encryption In Transit
**Traces to:** L1-006

All client connections to the database must use TLS 1.2 or higher. Unencrypted connections must be refused.

**Acceptance Criteria:**
1. Given the deployed database, when a client attempts to connect with `Encrypt=False`, then the connection must be refused or downgraded by the server's enforced setting.
2. Given a connecting client, when the connection is established, then the negotiated TLS version must be 1.2 or higher.

---

## L2-010: Encryption At Rest
**Traces to:** L1-006

All data files, log files, and backups must be encrypted at rest using Transparent Data Encryption (TDE), which is enabled by default on Azure SQL Database.

**Acceptance Criteria:**
1. Given the deployed database, when its TDE state is queried (`sys.dm_database_encryption_keys`), then encryption state must be `3` (encrypted).

---

## L2-011: Microsoft Entra ID Authentication
**Traces to:** L1-006

The database must authenticate clients using Microsoft Entra ID (Azure AD). SQL authentication accounts must not be created for application access; the application must connect using a managed identity.

**Acceptance Criteria:**
1. Given the deployed database, when its Entra admin is queried, then a Microsoft Entra admin (user or group) is configured.
2. Given the application's managed identity, when the application connects without any password or stored credential, then the connection must succeed and the authenticated principal must match the managed identity.
3. Given a client attempting to connect using SQL authentication for the application's data access, when the connection is attempted, then it must fail because no SQL login exists for that purpose.

**Traces to:** L1-008

3. Given the solution, when project references are inspected, then `BrainDump.Domain` does not transitively reference `BrainDump.Infrastructure` or `BrainDump.Api`.
4. Given the solution, when project references are inspected, then `BrainDump.Api` is the only project producing an executable host.

## L2-020: Responsive Breakpoint Behavior
**Traces to:** L1-010

The frontend must adapt its layout across the five viewport classes defined in L1-010 (XS <576, S ≥576, M ≥768, L ≥992, XL ≥1200). At every supported width, all primary navigation and editing actions must remain reachable without horizontal scrolling.

**Acceptance Criteria:**
1. Given the running application, when the viewport width is set to 360px (XS), then the section tree collapses behind a hamburger / drawer control and the editor occupies the full viewport width.
2. Given the running application, when the viewport width is set to 768px (M), then the section tree and editor are visible side-by-side or via an easily-toggled split.
3. Given the running application, when the viewport width is set to 1440px (XL), then the section tree, fact list, and editor are visible simultaneously without overlap.
4. Given any viewport width from 320px through 1920px, when the application renders, then no horizontal scrollbar appears on the document body and no interactive control is clipped beyond the viewport.

## L2-021: PKCE Authorization Code Flow (Frontend)
**Traces to:** L1-011

The frontend must implement OAuth 2.0 Authorization Code Flow with PKCE (RFC 7636) against an OpenID Connect identity provider. No client secret shall be embedded in the frontend bundle. Tokens must be requested with the `code_challenge_method=S256`.

**Acceptance Criteria:**
1. Given an unauthenticated user, when they navigate to the application, then they are redirected to the identity provider's authorization endpoint with `response_type=code`, `code_challenge`, and `code_challenge_method=S256` query parameters.
2. Given the user completes authentication and is returned to the application's redirect URI, when the frontend exchanges the authorization code for tokens, then the request to the token endpoint includes the original `code_verifier` and does not include a client secret.
3. Given the frontend bundle as built for production, when its source is inspected, then no client secret string is present.
4. Given a valid access token, when the frontend issues an API request, then the token is sent in an `Authorization: Bearer <token>` header.

---

## L2-022: JWT Bearer Validation (Backend)
**Traces to:** L1-011

The backend `BrainDump.Api` must validate incoming JWT bearer tokens against the configured identity provider's signing keys (issuer, audience, signature, and expiry). Endpoints that read or mutate sections or facts must require a valid token.

**Acceptance Criteria:**
1. Given a request to any section or fact endpoint without an `Authorization` header, when the backend processes it, then the response status is `401 Unauthorized`.
2. Given a request bearing a token signed by an untrusted key, when the backend validates it, then the response status is `401 Unauthorized`.
3. Given a request bearing a token whose `exp` claim has passed, when the backend validates it, then the response status is `401 Unauthorized`.
4. Given a request bearing a valid, unexpired token issued by the configured identity provider with the configured audience, when the backend processes it, then the request reaches the handler and is authorized to proceed.

## L2-023: Read-Tree Endpoint
**Traces to:** L1-002

The backend must expose an authenticated endpoint that returns the entire tree of sections and facts in a single response, suitable for the frontend to render the full editor on load.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `GET /api/tree`, then the response is `200 OK` with a JSON body containing two arrays: all sections (`id`, `parentId`, `title`, `position`) and all facts (`id`, `sectionId`, `text`, `position`).
2. Given the dataset specified in L2-006, when the endpoint is called, then the end-to-end response time at the p95 must be under 300 ms (allowing for serialization and network overhead beyond the 100 ms DB budget).

---

## L2-024: Section Mutation Endpoints
**Traces to:** L1-002

The backend must expose authenticated endpoints to create, update, and delete sections, each routed through a MediatR handler in `BrainDump.Application`.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `POST /api/sections` with a JSON body containing `parentId` (nullable), `title`, and `position`, then the response is `201 Created` with the new section's `id` and the row exists in the `section` table.
2. Given an authenticated client, when it issues `PUT /api/sections/{id}` with `parentId`, `title`, and `position`, then the response is `200 OK` (or `204 No Content`) and the row reflects the new values.
3. Given an authenticated client, when it issues `DELETE /api/sections/{id}`, then the response is `204 No Content` and the section, all descendant sections, and all their facts are removed (per L2-003).
4. Given a request to mutate a non-existent section ID, when the backend processes it, then the response status is `404 Not Found`.

---

## L2-025: Fact Mutation Endpoints
**Traces to:** L1-002

The backend must expose authenticated endpoints to create, update, and delete facts, each routed through a MediatR handler in `BrainDump.Application`.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `POST /api/facts` with a JSON body containing `sectionId`, `text`, and `position`, then the response is `201 Created` with the new fact's `id` and the row exists in the `fact` table.
2. Given an authenticated client, when it issues `PUT /api/facts/{id}` with `sectionId`, `text`, and `position`, then the response is `200 OK` (or `204 No Content`) and the row reflects the new values.
3. Given an authenticated client, when it issues `DELETE /api/facts/{id}`, then the response is `204 No Content` and the row is removed.
4. Given a request whose `text` is empty or whose `sectionId` does not exist, when the backend processes it, then the response status is `400 Bad Request` and no row is created or modified.

---

## L2-026: Sibling Reorder Endpoint
**Traces to:** L1-002, L1-005

The backend must expose a single authenticated endpoint that updates the `position` of a contiguous batch of siblings (either sections sharing a `parentId` or facts sharing a `sectionId`) in one transaction, used by the UI for drag-and-drop reordering.

**Acceptance Criteria:**
1. Given an authenticated client and three sibling facts with current positions 1, 2, 3, when it issues `POST /api/reorder` with a body specifying the new positions for each `id`, then the response is `200 OK` and a subsequent read returns the facts in the requested order.
2. Given a reorder request that would assign the same `position` value to two siblings, when the backend processes it, then the response status is `400 Bad Request` and no row is modified.
3. Given a reorder request that mixes IDs from different parents, when the backend processes it, then the response status is `400 Bad Request` and no row is modified.

## L2-031: Local Sign-In Endpoints (PKCE)
**Traces to:** L1-014, L1-011

The backend must expose a two-step OAuth 2.0 Authorization Code + PKCE flow at `POST /api/auth/authorize` and `POST /api/auth/token` that lets a developer exchange email + password (against the configured local-auth dev user) for a JWT access token. The flow mirrors the production PKCE flow specified in L1-011 so the frontend's authentication code path is identical in dev and production. Both endpoints must be reachable only when local sign-in is enabled by configuration.

**Acceptance Criteria:**
1. Given the backend is running with local sign-in enabled, when a client posts `{ "email", "password", "codeChallenge", "codeChallengeMethod": "S256" }` to `/api/auth/authorize` with the configured dev credentials, then the response is `200 OK` with a JSON body containing a `code` string. When the client then posts `{ "code", "codeVerifier", "grantType": "authorization_code" }` to `/api/auth/token` with a verifier whose SHA-256 base64url-encodes to the original `codeChallenge`, then the response is `200 OK` with a JSON body containing `accessToken`, `tokenType` (`"Bearer"`), and `expiresIn` (seconds).
2. Given the same configuration, when a client posts wrong credentials to `/api/auth/authorize`, then the response is `401 Unauthorized` and no code is returned.
3. Given the same configuration, when a client posts a malformed body (missing fields, empty values, unsupported `codeChallengeMethod`, or `/api/auth/token` with an unsupported `grantType` or a verifier that doesn't match the original challenge), then the response is `400 Bad Request` or `401 Unauthorized` as appropriate, and no token is returned.
4. Given a token returned by `/api/auth/token`, when a client uses it as a bearer token on `GET /api/tree`, then the request reaches the handler and is authorized to proceed (per L2-022 #4).
5. Given the backend is running with local sign-in disabled, when a client posts to either `/api/auth/authorize` or `/api/auth/token`, then the response is `404 Not Found` — not `401` — so the absence of the feature is unambiguous.

---

## L2-032: Local Sign-In Configuration and Production Safety
**Traces to:** L1-014

Local sign-in must be controlled by an explicit configuration flag (e.g., `Jwt:UseLocalAuth`), and the flag must default to `false`. The backend must refuse to start with local sign-in enabled in any environment whose ASP.NET Core environment name is not `Development`.

**Acceptance Criteria:**
1. Given the backend's default `appsettings.json` (production baseline), when configuration is loaded, then `Jwt:UseLocalAuth` is `false` (or absent, which resolves to `false`).
2. Given `appsettings.Development.json` ships with `Jwt:UseLocalAuth=true`, when the application is run with `ASPNETCORE_ENVIRONMENT=Development`, then it starts and the local sign-in endpoints (per L2-031) are reachable.
3. Given `Jwt:UseLocalAuth=true` is set in any non-Development environment, when the application starts, then startup must fail with a clear error message identifying the misconfiguration; the application must not begin serving requests.
4. Given local sign-in is enabled, when the backend issues an access token, then the token is signed with a key configured under `Jwt:LocalAuth` (separate from any production signing key) and is validated by the same JWT bearer pipeline used for production tokens.

## L2-033: Document Table Schema
**Traces to:** L1-015

The database must contain a `document` table with the following columns: `id` (INT IDENTITY, primary key), `folder_id` (INT, nullable, foreign key to `folder.id`), `title` (NVARCHAR(200), NOT NULL), `position` (INT, NOT NULL), `created_at` (DATETIME2, NOT NULL, defaulted to current UTC time on insert), and `updated_at` (DATETIME2, NOT NULL, set to current UTC time on insert and updated on any row change). A NULL `folder_id` denotes a document at workspace root.

**Acceptance Criteria:**
1. Given a fresh database, when migrations are applied, then the `document` table exists with exactly the columns `id`, `folder_id`, `title`, `position`, `created_at`, `updated_at` and the specified types and nullability.
2. Given the `document` table, when a row is inserted with `folder_id` referencing a non-existent folder, then the insert must fail with a foreign key violation.
3. Given a `document` row, when its `title` or `folder_id` is updated, then `updated_at` reflects a strictly later timestamp than the previous read of the same column.
4. Given the `document` table, when a non-clustered index on `(folder_id, position)` is queried, then it exists with the columns in that key order.

## L2-034: Section → Document Linkage
**Traces to:** L1-015 — amends L2-001

Each row in `section` must additionally carry a `document_id` column (INT, NOT NULL, foreign key to `document.id`). A section's `document_id` must equal its parent's `document_id`; root-level sections (`parent_id IS NULL`) carry the `document_id` of the document they belong to. Deleting a `document` must cascade to delete every `section` (and via L2-003, every `fact`) belonging to it.

**Acceptance Criteria:**
1. Given migrations applied to an existing database, when the `section` table is inspected, then the `document_id` column exists with the specified type, is `NOT NULL`, and is a foreign key to `document.id`.
2. Given a non-root section whose parent's `document_id` is `D`, when the row is inserted with a different `document_id`, then the insert must fail (CHECK or trigger).
3. Given a document with two sections and four facts under it, when the document row is deleted, then both sections and all four facts are removed in the same transaction.
4. Given the `section` table, when a non-clustered index on `(document_id, parent_id, position)` is queried, then it exists with the columns in that key order.

## L2-035: Document Mutation Endpoints
**Traces to:** L1-015

The backend must expose authenticated endpoints to create, update, and delete documents, each routed through a MediatR handler in `BrainDump.Application`.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `POST /api/documents` with a JSON body containing `folderId` (nullable), `title`, and `position`, then the response is `201 Created` with the new document's `id` and the row exists in the `document` table.
2. Given an authenticated client, when it issues `PUT /api/documents/{id}` with `folderId`, `title`, and `position`, then the response is `200 OK` (or `204 No Content`) and the row reflects the new values; `updated_at` advances per L2-033 #3.
3. Given an authenticated client, when it issues `DELETE /api/documents/{id}`, then the response is `204 No Content` and the document, all its sections, and all their facts are removed (per L2-034).
4. Given a request to mutate a non-existent document ID, when the backend processes it, then the response status is `404 Not Found`.

## L2-036: Folder Table Schema
**Traces to:** L1-016

The database must contain a `folder` table with the following columns: `id` (INT IDENTITY, primary key), `parent_id` (INT, nullable, foreign key to `folder.id`), `title` (NVARCHAR(200), NOT NULL), and `position` (INT, NOT NULL). A NULL `parent_id` denotes a root-level folder. Deleting a folder must cascade to delete its descendant folders, every document under it (and via L2-034, every section and fact under those documents), in a single atomic operation.

**Acceptance Criteria:**
1. Given a fresh database, when migrations are applied, then the `folder` table exists with exactly the columns `id`, `parent_id`, `title`, `position` and the specified types and nullability.
2. Given the `folder` table, when a row is inserted with `parent_id` referencing a non-existent folder, then the insert must fail with a foreign key violation.
3. Given a folder with one child folder and one document containing one section and two facts, when the parent folder is deleted, then the descendant folder, the document, the section, and both facts no longer exist after the operation commits.
4. Given the `folder` table, when a non-clustered index on `(parent_id, position)` is queried, then it exists with the columns in that key order.

## L2-037: Folder Mutation + Move Endpoints
**Traces to:** L1-016

The backend must expose authenticated endpoints to create, update, and delete folders, plus an endpoint that reparents (moves) a folder or document. Reparenting must validate that the target parent is not equal to or a descendant of the moved node (no cycles).

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `POST /api/folders` with `parentId` (nullable), `title`, and `position`, then the response is `201 Created` with the new folder's `id`.
2. Given an authenticated client, when it issues `PUT /api/folders/{id}` with `parentId`, `title`, and `position`, then the response is `200 OK` (or `204 No Content`) and the row reflects the new values.
3. Given an authenticated client, when it issues `DELETE /api/folders/{id}`, then the response is `204 No Content` and L2-036 #3 cascade applies.
4. Given an authenticated client, when it issues `POST /api/move` with a JSON body containing `kind` (`"folder"` or `"document"`), `id`, `targetParentId` (nullable), and `position`, then the response is `200 OK` and a subsequent read shows the node under the new parent at the requested position.
5. Given a move request that would place a folder inside itself or one of its descendants, when the backend processes it, then the response status is `400 Bad Request` and no row is modified.

## L2-038: Document-Scoped Tree Endpoint
**Traces to:** L1-015 — amends L2-023

Once L1-015 is in effect, the tree-read endpoint must be scoped to a single document. The backend must expose `GET /api/documents/{documentId}/tree` returning every section and fact belonging to that document. The original `GET /api/tree` endpoint is superseded; clients must supply a `documentId`.

**Acceptance Criteria:**
1. Given an authenticated client and an existing document with id `D`, when it issues `GET /api/documents/D/tree`, then the response is `200 OK` with a JSON body containing two arrays: all sections and all facts whose `document_id = D`. Section/fact shapes match L2-023 #1.
2. Given the dataset specified in L2-006 partitioned across 100 documents (so each document carries ~10 sections and ~100 facts), when the endpoint is called for a single document, then the end-to-end response time at the p95 must be under 200 ms.
3. Given a request for a non-existent document ID, when the backend processes it, then the response status is `404 Not Found`.

## L2-039: Open Tabs Persistence
**Traces to:** L1-017

The frontend must persist the user's set of open document tabs and the active tab so they survive a browser reload and a fresh sign-in on a different machine. Storage must be server-side (per-user) so the workspace follows the user across devices.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `PUT /api/tabs` with a JSON body containing an ordered array of `documentId`s and an `activeIndex`, then the response is `200 OK` and the state is persisted for the calling user.
2. Given a user with previously-persisted tab state, when they sign in on any device and the frontend issues `GET /api/tabs`, then the response is `200 OK` with the persisted ordered array and `activeIndex`.
3. Given a tab state that references a document the user can no longer access (deleted), when the frontend reads it, then the deleted document is silently filtered from the response and `activeIndex` is clamped to a valid value.

## L2-040: Split View Behavior
**Traces to:** L1-017

The frontend must support displaying two open documents side-by-side in a single editor pane. Each pane has its own active tab; switching tabs in one pane must not affect the other. The split must be invocable from a control on the tab bar.

**Acceptance Criteria:**
1. Given a single editor pane with one or more open tabs, when the user activates the "Split" control, then the editor area divides into two panes side-by-side and the previously-active document is shown in the new (right) pane.
2. Given a split editor with two panes, when the user closes the last tab in either pane, then the editor returns to a single-pane layout and the remaining pane fills the editor area.
3. Given a split editor, when the user clicks a tab in the left pane, then the right pane's active tab is unchanged and vice-versa.
4. Given a split editor at viewport width below 992px (per L1-010 medium breakpoint), when the layout adapts, then the split collapses to a single pane and the previously-secondary pane is reachable via the tab bar.

## L2-043: Document Reference Extraction
**Traces to:** L1-019

The backend must persist outgoing references between documents. A reference is created whenever the body of one document contains a wiki-link to another document — syntax `[[document-title]]` resolved by exact-match title, or `[[id:N]]` resolved by id. References are stored in a `document_link` table (`source_document_id` INT NOT NULL FK, `target_document_id` INT NOT NULL FK, primary key `(source_document_id, target_document_id)`). Reference extraction must run synchronously inside the same transaction as the document mutation that produced it.

**Acceptance Criteria:**
1. Given a fresh database, when migrations are applied, then the `document_link` table exists with the specified columns, types, and primary key.
2. Given an existing document A and a fact in document B whose text contains `[[A-title]]`, when document B is created or its content is updated, then a `document_link(source=B, target=A)` row exists after the transaction commits.
3. Given a reference link to a non-existent document title, when the document is saved, then the save still succeeds and no `document_link` row is produced for the unresolved reference.
4. Given a document with outgoing references, when the document is deleted, then every `document_link` row where `source_document_id` or `target_document_id` equals the deleted id is removed by `ON DELETE CASCADE`.

## L2-045: Global Search Endpoint
**Traces to:** L1-020

The backend must expose `GET /api/search?q={query}&limit={n}` returning matches across document titles, section titles, fact text, and labels. Results must include a snippet showing the matched span and a stable score for ranking.

**Acceptance Criteria:**
1. Given an authenticated client, when it issues `GET /api/search?q=brain-dump`, then the response is `200 OK` with a JSON array of result objects shaped `{ documentId, title, snippet, matchedField, score }`, sorted by `score` descending.

3. Given a query that matches no documents, when the backend processes it, then the response is `200 OK` with an empty array.

## L2-047: Recently Viewed Storage and Endpoint
**Traces to:** L1-021

The database must record per-user document views in a `user_document_view` table (`user_id` INT NOT NULL FK, `document_id` INT NOT NULL FK, `viewed_at` DATETIME2 NOT NULL, primary key `(user_id, document_id)`). ## L2-048: Page Template Storage
**Traces to:** L1-022

The database must contain a `template` table (`id` INT IDENTITY primary key, `title` NVARCHAR(200) NOT NULL, `description` NVARCHAR(1000) nullable, `created_at` DATETIME2 NOT NULL) and parallel `template_section` and `template_fact` tables that mirror the schema of `section` and `fact` (per L2-001, L2-002) except that they reference `template_id` instead of `document_id` and inherit no relationship to live `section`/`fact` rows. 3. Given the `template` table, when its rows are read, then no row references any live `document`, `section`, or `fact` row (templates and live content are isolated).

## L2-049: Document Instantiation From Template
**Traces to:** L1-022

The backend must expose an endpoint that creates a new document by deep-copying a template's section/fact tree.

**Acceptance Criteria:**
1. Given an authenticated client and a template `T`, when it issues `POST /api/documents` with a JSON body `{ folderId, title, position, templateId: T }`, then the response is `201 Created` and the resulting document carries a fresh copy of every `template_section` and `template_fact` row in `T`, with new section and fact ids.
2. Given a document created from a template, when the underlying template is later edited, then the document's content must not change (independent copies).
3. Given a `templateId` that does not exist, when the backend processes the request, then the response is `400 Bad Request` and no document is created.
4. Given a `POST /api/documents` request without `templateId`, when the backend processes it (per L2-035 #1), then the document is created empty (no sections, no facts).

# L1 - High-Level Requirements

> **Scope:** Full brain-dump system — frontend, backend, and database. Markdown import/export between the system and the canonical `brain-dump.md` file is out of scope for this document and will be specified separately.

## L1-001: Hierarchical Fact Storage
The system must persist a tree of named sections, each containing an ordered list of facts (free-text declarative statements). The structure must directly mirror the source-of-truth markdown document so that any tree state can be rendered back to equivalent markdown.

## L1-002: Full CRUD Over Sections and Facts
The system must support create, read, update, and delete operations for both sections and facts, including reparenting a section under a different parent and reordering siblings within their parent.

## L1-003: Interactive-Grade Performance
The database must return any read or write operation fast enough to feel synchronous in a UI driven by a single editor. A full-tree read and any single-row write must complete within strict latency budgets defined in L2.

## L1-004: Azure SQL Database Deployment
The system must run on Microsoft Azure SQL Database as a managed PaaS service, using a tier and configuration that is always-on (no cold-start delay on first interactive request) while remaining cost-appropriate for a single-editor workload.

## L1-005: Referential and Ordering Integrity
The schema must enforce, at the database level, that every fact belongs to an existing section, that deleting a section removes its descendants and facts atomically, and that sibling ordering is preserved deterministically across reads.

## L1-006: Managed Security Defaults
The database must require authenticated access using Microsoft Entra ID, encrypt all data in transit with TLS, and encrypt all data at rest. No password-based SQL authentication shall be used in deployed environments.

## L1-007: Frontend Application Stack
The user-facing application must be an Angular single-page application built with Angular Material as the component library and the Microsoft Monaco editor (https://github.com/microsoft/monaco-editor) embedded for editing fact text. The frontend must be organized as an Angular Workspace.

## L1-008: Backend Application Architecture
The backend must be a .NET application organized using Clean Architecture (Domain, Application, Infrastructure, Api) with MediatR as the request/handler pipeline mediator. MediatR must be pinned to its last open-source version (v12.x under the Apache 2.0 license); MediatR v13 and later require a commercial license and must not be used.

## L1-009: Repository Layout
The repository must contain a top-level `frontend/` folder holding the Angular workspace and a top-level `backend/` folder holding the .NET solution. The backend solution file must be `backend/BrainDump.sln` in the legacy (non-SDK-style) `.sln` format, with `backend/src/` containing project source and `backend/tests/` containing test projects.

## L1-010: Responsive Design
The frontend must render and remain fully usable across all standard viewport sizes: extra small (<576px), small (≥576px), medium (≥768px), large (≥992px), and extra large (≥1200px). No horizontal scrolling, content clipping, or unreachable controls shall occur at any supported viewport size.

## L1-011: User Authentication via PKCE
The frontend must authenticate users through OAuth 2.0 / OpenID Connect Authorization Code Flow with PKCE (RFC 7636). The backend must accept and validate access tokens issued by the configured identity provider; unauthenticated requests to protected endpoints must be rejected.

## L1-012: Containerized Local Development Environment
The solution must support local development via Docker and Docker Compose. A single `docker compose up` from the repository root must bring up every dependency required to run the application end-to-end on a developer workstation, with no manual installation of databases or other infrastructure beyond Docker itself.

## L1-013: Microsoft SQL Server in Docker for Local Development
For local development, the system must use Microsoft SQL Server running in a Docker container as its database. The container must be defined in the local Docker Compose file, must persist data across restarts via a named volume, and must expose its port to the host so the backend can connect from inside or outside the Compose network. SQL authentication may be used locally — this is the one explicit exception to L1-006, which continues to apply to all deployed environments.

## L1-014: Local Sign-In for Development
The backend must implement a local sign-in mechanism that allows developers to authenticate against the API without depending on an external identity provider. The local sign-in must produce credentials that satisfy the same JWT bearer validation pipeline used in production (L1-011), so protected endpoints behave identically under both authentication modes. Local sign-in must be gated by configuration and must be disabled by default in production builds.
